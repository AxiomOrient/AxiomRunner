# `crates/infra` 아키텍처 분석 리포트

현재 `crates/infra` 크레이트에 대한 구조적 타당성, 도메인 경계 분리, 그리고 에러 처리 모델링에 관한 분석 결과입니다.

## 1. Data Model

### Problem (문제 정의)

`crates/infra` 모듈의 아키텍처적 역할 분담 충실도, 크레이트 세분화의 적절성, 그리고 헥사고날 아키텍처 상의 도메인 경계 유지 여부에 대한 검증.

### Evidence (증거)

1. **나노 크레이트(Nano-crate)**: `crates/infra` 폴더 내부에는 실제 인프라스트럭처 구현체(DB Pool, HTTP Client 등)가 전혀 존재하지 않으며, 단지 `error.rs` 파일 하나(100줄 미만)만 존재합니다.
2. **도메인 개념 유출**: `error.rs`에 정의된 `InfraError` 열거형 내부에 `InvalidInput`, `Unavailable`, `Timeout` 외에 `PolicyViolation` 이라는 도메인 정책 관련 에러 타입이 포함되어 있습니다.
3. **정적 재시도 매핑**: `InfraError`에 내장된 `retry_class()` 메서드를 통해 에러 타입 자체가 자신이 `Retryable`인지 `NonRetryable`인지 정적으로 반환하고 있습니다.

### Hypotheses (가설 및 검증)

- **가설 1 (가정 오류 / 크레이트 파편화)**: `crates/infra`는 실제 인프라 로직을 담고 있지 않으며, 오직 외곽 어댑터 계층에서 공통으로 쓸 에러 타입 하나를 위해 억지로 분리된 과도하게 파편화된 크레이트이다.
  - _검증 절차_: `crates/adapters` 영역 등에서 `axonrunner_infra`의 사용처를 검색(`grep`)해 봅니다. 에러 타입(`InfraError`) 외에 어떤 인프라 통신 추상화나 유틸리티도 참조되지 않는다면 이 가설은 참입니다.
- **가설 2 (계약 위반 / 도메인 침범)**: `InfraError::PolicyViolation`은 헥사고날 아키텍처의 명백한 계약 위반이다. 인프라스트럭처 계층의 에러는 '네트워크 타임아웃'이나 '접근 거부/연결 실패'여야지, 비즈니스 도메인의 '정책 위반(Policy)' 개념을 알아서는 안 된다.
  - _검증 절차_: 외부 API에서 권한 없음(Forbidden) 오류가 났을 때 처리 로직을 추적해 봅니다. 단순 접속 실패가 아니라 인프라 레이어가 이를 `PolicyViolation`으로 해석하여 위로 던지도록 강제되어 있다면, 인프라가 비즈니스 로직을 판단하고 있는 것이므로 이 가설은 참입니다.
- **가설 3 (컨텍스트 증거 누락)**: 에러 자체가 `.classify()`로 `RetryClass`를 반환하는 것은, 호출자(Caller)의 컨텍스트(예: 이미 재시도 한도를 초과했거나 외부 인터럽트가 발생한 상황)를 무시하는 '컨텍스트 증거 누락'의 설계 오류다.
  - _검증 절차_: 전체 시스템 타임아웃이 1ms 남은 상황에서 `InfraError::Timeout`이 발생하도록 시뮬레이션 합니다. 호출자의 남은 시간 컨텍스트와 무관하게 에러 객체 자체는 무조건 `Retryable`을 반환하게 되므로 이 가설은 참입니다.

### Options (대안)

- **대안 A (현상 유지 / Status Quo)**
  - 에러 타입 전용 크레이트로 내버려 둡니다.
  - _트레이드오프_: 수정 비용은 0이지만, 계속해서 워크스페이스에 불필요한 나노 크레이트가 차지하며 코어 도메인 개념(`PolicyViolation`)이 인프라에 오염된 채로 남습니다.
- **대안 B (`adapters` 로 통폐합)**
  - `error.rs` 하나 뿐인 이 크레이트를 삭제하고, 의존성의 주체인 `crates/adapters/src/` 내부의 공통 파일로 에러 타입을 흡수시킵니다.
  - _트레이드오프_: 크레이트 파편화는 막을 수 있으나, 향후 DB Connection Pool이나 Telemetry 로깅 같은 진짜 공통 '인프라' 코드를 배치할 전용 공간이 사라지게 됩니다.
- **대안 C (인프라 경계 순수화 및 유지 방어)**
  - `crates/infra` 크레이트 구조는 그대로 유지하되, `InfraError` 내부에서 도메인 지식인 `PolicyViolation`을 삭제하여 의존성 오염(Contract Violation)을 해결합니다. 또한 `RetryClass` 반환 로직을 에러 객체 외부(실행 컨텍스트나 Policy)로 분리합니다.
  - _트레이드오프_: 가장 이상적인 아키텍처 경계를 확보하며, 향후 범용 인프라 코드가 들어올 자리를 무사히 비워둘 수 있습니다.

### Decision (결정 및 권고사항)

- **권고안**: **대안 C (인프라 경계 순수화 및 유지 방어)**
- **채택 이유**: 단 100줄짜리 극소형 크레이트이더라도 향후 확장을 고려해 워크스페이스 구조를 유지하는 것이 낫습니다. 단, 인프라스트럭처가 비즈니스 도메인 용어(`Policy`)를 아는 것은 심각한 헥사고날 규칙 위반(Contract Violation)이므로 이를 즉시 제거하여 인프라의 순수성(`Timeout`, `Unavailable` 등)만 남겨두는 설계가 가장 타당합니다.

## 2. Immediate Next Action

- 다음 스프린트 또는 리팩토링 단계에서 `crates/infra/src/error.rs` 안의 `PolicyViolation` 변형을 제거하고, 이를 호출하는 어댑터 단에서 도메인 에러(Core 의존성)로 변환하도록 매핑 로직을 분리하는 패치를 계획합니다. (현재 단계에서는 코드 수정을 진행하지 않습니다)
